## Queue redesign — ONE design for #30/#31/#32 (risk note: all three edit mcp-server.mjs :290, :465-497, :122-144, :291-328 — spec as one redesign, landed in two stages with three verifications, not three patches)

Target state. `spawnQueues` becomes per-**workspace-directory** state: `wsOf(key) → { writerTail: Promise, writersPending: int }`, key = `dirname(resolve(graphPath))` for graph tools, `resolve(out)` for map, `dirname(resolve(queueFrom(args)))` for keyed graphless tools. One shared `runChild(id, spec)` wrapper owns spawn/timeout/settle/trace/pendingAsync for every child (map, autoRefresh, tools) — #34's inflight map and the trace hook plug into it once.

Invariants (each gets a scenario or unit test):
- **I1** Writers on one workspace never overlap; FIFO among writers (chain on `writerTail`).
- **I2** A reader never starts before an earlier-queued writer on its workspace (reader awaits the `writerTail` snapshot taken at enqueue time).
- **I3** Writers do NOT wait for earlier readers. Safe because every artifact write is rename-atomic (`lib/cli.mjs atomicWrite`): a mid-read reader keeps the old inode — its answer reflects its enqueue-time state either way. (Behavior change vs today's full serialization; document in the :465 comment.)
- **I4** Readers run concurrently under a **global cap of 3 children** (FIFO waiter queue; constant `READER_CAP = 3`; setting it to 1 restores today's serialization — the rollback lever).
- **I5** Every job settles exactly once — reply | errResult | cancel-suppressed — and always releases its queue slot + `asyncDone()`, so stdin-close drain (:287-289, :524) never hangs on a suppressed reply.
- **I6** In-process fast paths never queue, EXCEPT the #33 diff fast path, which awaits its after-workspace `writerTail` first (else #33 would reopen #30).

Writer/reader classification on TOOLS (`kind: 'writer'` field; default reader): **writers** = `codeweb_refresh`, `codeweb_annotate` (mutates annotations.json + graph consumers re-read), `codeweb_map`, plus the internal autoRefresh child. **Readers** = every other spawned tool and every fast-path fallback spawn (diff, find_similar, review, risk, hotspots, break_cycles, deadcode, campaign, reading_order, simulate, codemod, placement, fitness, stats, query/brief/find/explain/context fallbacks).

Observability for BDD-by-mechanism: `CODEWEB_MCP_TRACE=1` → one NDJSON event per line on **stderr** ({ev:'start'|'end'|'kill'|'skip-autorefresh', id, tool, ws, pid}). Default off; stdout purity untouched (update the :14 "here: none" comment). Scenario tests assert event interleavings, not wall-clock.

## #34 — cancellation + malformed frames — Low/S

- **T-34.1** id→child map: `runChild` registers `inflight.set(id, {kill, cancelled})`; deleted on settle. In `handle()` (:500-506), BEFORE the id-less drop: `if (method === 'notifications/cancelled') { const j = inflight.get(params?.requestId); if (j) { j.cancelled = true; j.kill(); } return; }` — kill (SIGKILL) + **suppress the reply** (I5: still release queue/slot/asyncDone). A cancelled job still queued-not-started never spawns (checked when the chain reaches it). Unknown/settled requestId → ignore (MCP: MAY ignore); `initialize` and fast-path answers are un-cancellable (already settled).
- **T-34.2** Non-object frames (numbers/strings/null/bools after parse) → `fail(null, -32600, 'Invalid Request')` instead of silent drop.
- **T-34.3** Batch arrays — decision, verified: `PROTOCOLS` (:43) negotiates `2024-11-05 | 2025-03-26 | 2025-06-18` (echo-if-supported). MCP changelog: batching added in 2025-03-26 ("MUST support receiving"), **removed in 2025-06-18**; 2024-11-05 rides base JSON-RPC 2.0 where batching exists. So pure rejection is strictly in-spec only for 2025-06-18. **Decision: reject every array with one `-32600` error (`id:null`, message: 'JSON-RPC batching not supported; send one request per line'); keep PROTOCOLS unchanged.** Rationale: implementing batch response collection couples badly with async children + cancel-suppression (a cancelled member would hold the response array open forever); narrowing PROTOCOLS to 2025-06-18 makes 2024-11-05-era clients disconnect at initialize (worse than the bug). The documented tradeoff: a batching 2025-03-26 client now gets an immediate visible error instead of a silent hang; no known stdio MCP client batches (bench/all.mjs:114 pins 2024-11-05 but sends NDJSON singles). Reviewer: veto here means implementing collection — say so explicitly.
- Tests: scenario **S4**; unit: `[]`, `[{...}]`, `42`, `"x"`, `null` each → -32600 and a following ping answers.
- Criteria: cancelled `codeweb_map` → child killed (trace `kill` + no graph.json appears), no reply for that id, server drains and exits 0.
