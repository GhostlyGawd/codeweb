## Queue redesign — ONE design for #30/#31/#32 (risk note: all three edit mcp-server.mjs :290, :465-497, :122-144, :291-328 — spec as one redesign, landed in two stages with three verifications, not three patches)

Target state. `spawnQueues` becomes per-**workspace-directory** state: `wsOf(key) → { writerTail: Promise, writersPending: int, readersInFlight: Set<Promise> }`, key = `dirname(resolve(graphPath))` for graph tools (note: today's key is the graph FILE path — the identity change to the dir is what lets refresh/diff/map collide), `resolve(out)` for map, `dirname(resolve(queueFrom(args)))` for keyed graphless tools. One shared `runChild(id, spec)` wrapper owns spawn/timeout/settle-once/trace/pendingAsync/inflight-registration for every child (map, autoRefresh, tools) — #34's inflight map and the trace hook plug into it once.

Invariants (each gets a scenario or unit test):
- **I1** Writers on one workspace never overlap; FIFO among writers (chain on `writerTail`).
- **I2** A reader never starts before an earlier-queued writer on its workspace (reader awaits the `writerTail` snapshot taken at enqueue time). Fairness corollary: once a writer is queued, every later reader on that workspace waits behind it — continuous reader arrival cannot starve a writer.
- **I3** *(rewritten — the conservative rule)* A writer also waits for every reader **enqueued before it** on its workspace: its chain link is the join of `writerTail` and a snapshot of `readersInFlight` at enqueue. Rationale: rename-atomicity makes any SINGLE read safe, but spawned readers make MULTIPLE workspace reads (find_similar reads graph.json then similar-index.json, whose loader stamps against a *later* statSync of the graph — similar-index.mjs:34-41): a writer landing between two reads hands the reader a mixed old/new state that today's full serialization makes impossible. Preserving today's linearization is the contract; results carry no graph-identity stamp, so the reader has no way to label what it read. Cost is bounded: the writer delays by at most the in-flight readers' runtime (worst case `SPAWN_TIMEOUT_MS` — exactly today's behavior, where it queued behind them anyway). #32's measured win (reader-reader overlap) is untouched. The aggressive variant (writers skip readers) is REJECTED, not deferred: adopting it later requires per-result graph-stamp labeling first.
- **I4** Readers run concurrently under a **global cap of 3 children** (FIFO waiter queue; constant `READER_CAP = 3`; setting it to 1 restores today's serialization — the rollback lever). Ordering is load-bearing: a reader acquires its slot only AFTER its I2 writerTail wait resolves, and holds it only while its child runs — a reader blocked on a writer holds no slot, so one workspace's queued readers can never starve or deadlock other workspaces.
- **I5** Every job settles exactly once — reply | errResult | cancel-suppressed — and always releases its reader slot, decrements `writersPending` (if a writer), deletes its `inflight` entry, and calls `asyncDone()`, on every path including cancel-suppressed and cancelled-before-spawn, so stdin-close drain (:287-289, :524) never hangs. Settle-once must guard the `error`+`close` DOUBLE-fire (Node docs: 'exit'/'close' may or may not follow 'error' — guard both): the spawned-tool path has the `settled` flag today, autoRefresh has `done()`'s, but **handleMap has neither** — a spawn failure there double-replies and double-decrements `pendingAsync` (latent today; fixed by routing through `runChild`). Unit: unspawnable bin (ENOENT) → exactly one reply, `pendingAsync` balanced, exit 0 on close.
- **I6** In-process fast paths never queue, EXCEPT the #33 diff fast path, which awaits its after-workspace `writerTail` first (else #33 would reopen #30). It is deliberately NOT registered in `readersInFlight` (it makes one read of the live workspace — the after side — and single reads are rename-atomic-safe; the before side is a caller-supplied snapshot path).
- **I7** `writersPending` increments synchronously at enqueue time (before any await) and decrements at settle — so autoRefresh's dedup check (T-31.2) reads it race-free within the same readline drain. A queued READER never suppresses autoRefresh (readers cannot supply freshness); only `writersPending > 0` does.

Writer/reader classification on TOOLS (`kind: 'writer'` field; default reader): **writers** = `codeweb_refresh`, `codeweb_annotate` (mutates annotations.json + graph consumers re-read), `codeweb_map`, plus the internal autoRefresh child. **Readers** = every other spawned tool and every fast-path fallback spawn (diff, find_similar, review, risk, hotspots, break_cycles, deadcode, campaign, reading_order, simulate, codemod, placement, fitness, stats, query/brief/find/explain/context fallbacks).

Observability for BDD-by-mechanism: `CODEWEB_MCP_TRACE=1` → one NDJSON event per line on **stderr** ({ev:'start'|'end'|'kill'|'skip-autorefresh', id, tool, ws, pid; kill carries reason:'cancel'|'timeout'}). Default off with ZERO stderr writes when unset (update the :14 "here: none" comment: stdout purity untouched; stderr stays silent by default). Stderr is free-form in MCP stdio so trace cannot corrupt the protocol, but every trace write is try/catch-guarded (a client may close stderr — the #29 lesson applies to all three fds). Scenario tests assert event interleavings, not wall-clock.

## #34 — cancellation + malformed frames + batches — Low/S

- **T-34.1** id→child map: `runChild` registers `inflight.set(id, {kill, cancelled})`; deleted on settle. In `handle()` (:500-506), BEFORE the id-less drop: `if (method === 'notifications/cancelled') { const j = inflight.get(params?.requestId); if (j) { j.cancelled = true; j.kill(); } return; }` — kill (SIGKILL) + **suppress the reply** (I5: still release slot/writersPending/asyncDone). A cancelled job still queued-not-started never spawns (checked when the chain reaches it; releases everything per I5). Unknown/settled requestId → ignore (MCP: MAY ignore). Synchronous fast-path answers are un-cancellable (settled within the same drain). Id-reuse note: cancellation targets the CURRENT inflight holder of an id; JSON-RPC requires uniqueness only among outstanding requests, so a client that reuses a settled id accepts that a late cancel for the old use may hit the new one — documented, not defended.
- **T-34.2** Non-object frames (numbers/strings/null/bools after parse) → `fail(null, -32600, 'Invalid Request')` instead of silent drop.
- **T-34.3** Batch arrays — decision, verified: `PROTOCOLS` (:43) negotiates `2024-11-05 | 2025-03-26 | 2025-06-18`; batching is in-spec for the first ("MUST support receiving" in 2025-03-26; base JSON-RPC 2.0 for 2024-11-05) and removed in 2025-06-18. **Decision: minimal per-member fan-out.** A non-empty array → each member is processed exactly as if it had arrived alone (object member → normal handling under its own id; non-object member → its own `-32600` `id:null` line, per JSON-RPC's batch examples; member notifications → silence; nested array member → `-32600`); responses are emitted as individual NDJSON lines in settle order, **not collected into a response array**. Empty array `[]` → single `fail(null, -32600, 'Invalid Request')` (the JSON-RPC-mandated shape for an empty batch). PROTOCOLS unchanged. Rationale, explicitly adjudicated: (i) pure rejection (`-32600` for every array) fixes the silent hang but violates 2025-03-26's MUST for well-formed batches — rejected; (ii) full array-collection is NOT ~15 lines here: `reply`/`errResult`/`send` are invoked from a dozen async settle sites (children, timers, cancel paths), so a collector threads through every one, and a cancel-suppressed member (MCP: a cancelled request gets NO response) would hold the response array open forever — a structural contradiction, rejected; (iii) fan-out fixes the hang for every by-id-correlating client, has no coupling with async/cancel (a cancelled member behaves exactly like an unbatched cancel), and costs the documented deviation that responses are line-framed rather than array-framed — strict array-collecting clients remain unsupported (none known: bench/all.mjs:114 pins 2024-11-05 and sends NDJSON singles). No batch-size cap: members flow through the same queue/READER_CAP/pendingAsync bounds as unbatched lines.
- Tests: scenario **S4** + **S7**; units: `[]` → one -32600 id:null; `42`, `"x"`, `null`, `true` → -32600 each; `[1,2]` → two -32600 lines; `[{ping id:9},{unknown-tool id:10}]` → ping reply for 9 AND -32602 for 10; `[{notification}]` → no response but a following ping answers; cancel-during-diff-fast-path → no reply for that id, ping answers.
