## Finding #6 — one 60 s sequential property test floors the suite

**T-6.1 — async spawn helper · hardened.** Files: `tests/helpers.mjs`.
Approach: export `runNodeAsync(scriptPath, args, {env, cwd})` — `child_process.execFile` wrapped in a promise, same contract as `runNode` (resolves `{status, stdout, stderr}`, never rejects; `status` from `error.code ?? 1` on failure; `maxBuffer: 1 << 28`) — INCLUDING the env merge `{...process.env, ...env}` exactly as runNode does at helpers.mjs:23 (dropping the merge strands children without PATH/CODEWEB_* vars). Measured basis for the design: node:test runs top-level `test()`s in one file SEQUENTIALLY (4×400 ms slept 1.8 s), and `{concurrency}` subtests overlap only when bodies yield — spawnSync bodies serialized even under concurrency 4 (1.98 s) while async bodies overlapped (0.58 s). So the audit's "per-trial test()s run concurrently" shape is rejected: concurrency requires subtests + async spawn. (Independently re-verified by spec review: 4 execFile subtest bodies of 400 ms under parent `{concurrency: 4}` → 455 ms wall, this container.)
TDD (RED first): unit test in a tests file: `runNodeAsync` on a `process.exit(3)`-style tmp script returns status 3 + captured stderr; two awaited-in-parallel 400 ms sleep-scripts overlap — assert wall < 750 ms (generous margin; if CI jitters, loosen upward, never tighten).
Success: helper green; no behavior change elsewhere. Risk: none (additive).
